# Deployment (GCP)

This folder was generated by the AIDE deployment phase. **AIDE only generated
these files — it never ran any `gcloud`/`terraform` commands.**

## Cairn is a desktop application, not a cloud service

Cairn is a local-first cross-platform Electron desktop app packaged into
Windows / macOS / Linux installers by `.github/workflows/package.yml`. There is
**no server-side compute** — no Cloud Run, no Cloud Functions, no VMs. Users
run the binary on their own machine.

What this `deploy/` folder DOES provision on GCP is the small amount of cloud
infrastructure a distributed desktop app still benefits from:

- A **Cloud Storage bucket per environment** to host the built installer
  artifacts (`.exe`, `.msi`, `.dmg`, `.AppImage`) and their `latest.yml`
  manifests. Downloads and (future) auto-update pull from these buckets.
- A **least-privilege runtime service account** with `objectCreator` on that
  bucket only — so the packaging pipeline can upload installers without any
  wider project rights.

Everything else (auto-update endpoint, code-signing, notarization) is
deliberately deferred per the spec's hardening backlog and is a TODO in
`electron-builder.yml`.

## Layout

- `terraform/modules/` — reusable, vetted Terraform modules (`storage-bucket`,
  `iam`).
- `terraform/environments/{staging,production}/` — per-environment wiring.
  Each has a `*.tfvars.example` (committed, **documentation only** — CI does NOT
  use it).
- `bootstrap/` — one-time setup (Terraform state bucket, deploy service
  account, Workload Identity Federation). Run by a human with project admin.
- `../.github/workflows/deploy-*.yml` — CI: staging on PR, production gated on
  a `v*` tag with required reviewer approval.

## Configuration — no committed tfvars

Terraform inputs (`project_id`, `region`, `env`) are supplied at deploy time by
the GitHub Actions workflows via `TF_VAR_*` environment variables, sourced from
**repo variables** (Settings → Secrets and variables → Actions → Variables).
There is intentionally no committed `*.tfvars` — the repo variables are the
single source of truth. The `*.tfvars.example` files exist only so you can
`cp staging.tfvars.example staging.tfvars` for a local `terraform plan`
(real `*.tfvars` is gitignored).

## One-time setup (per GCP project — staging AND production)

1. `cd deploy/bootstrap`
2. `terraform init && terraform apply` with
   `-var project_id=… -var region=… -var github_repo=owner/repo -var state_bucket=…`
3. Note the `wif_provider` and `deploy_sa_email` outputs.
4. In GitHub repo settings add **variables** (Settings → Secrets and variables →
   Actions → Variables):
   - `GCP_WIF_PROVIDER` — the `wif_provider` output above
   - `GCP_DEPLOY_SA` — the `deploy_sa_email` output above
   - `GCP_REGION` — e.g. `europe-west1`
   - `GCP_PROJECT_STAGING` — staging project ID
   - `GCP_PROJECT_PRODUCTION` — production project ID
   - `GCP_ARTIFACT_REPO` — unused for Cairn (no container images) but reserved
     for parity with the AIDE template. Set to any non-empty placeholder.
   - Protect the `production` environment with a required reviewer.
5. Fill the `bucket`/`prefix` in each `environments/*/backend.tf` with the
   state-bucket name you created above.

## Uploading installers to the bucket

After `.github/workflows/package.yml` produces per-OS installers, upload them
to the environment bucket manually or via a follow-up workflow step:

```bash
gcloud storage cp dist/installers/* \
  gs://<project_id>-cairn-releases-<env>/v<version>/ \
  --project <project_id>
```

This is intentionally NOT automated in this pass — code signing and update
manifests must land first (see the ADR hardening backlog).

## Secrets

Cairn is local-first and reads all runtime secrets from the **OS keychain**
(planned adapter: `@napi-rs/keyring`) — never from environment variables and
never from GCP. As a result no `secret` module is provisioned here.

If a future feature (e.g. AI news pull key held server-side, a Claude proxy)
requires a cloud secret, add the `secret` module then and wire it via
`secret_env` on whatever compute is introduced.

## Deploy

- **Staging:** open a PR → `deploy-staging.yml` applies the staging bucket +
  IAM automatically.
- **Production:** push a `v*` tag (or run the workflow manually) → applies
  after the required approval on the `production` environment.
